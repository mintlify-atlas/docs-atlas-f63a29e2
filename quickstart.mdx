Roxy is a self-hosted proxy server that runs on Heroku to unblock websites. This guide will walk you through deploying your own Roxy instance in just a few minutes.

## Prerequisites

Before you begin, make sure you have:

- A [Heroku account](https://signup.heroku.com/) (free tier works)
- Git installed on your machine (for manual deployment)
- [Heroku CLI](https://devcenter.heroku.com/articles/heroku-cli) installed (for manual deployment)

## One-click deployment

The fastest way to get Roxy up and running is using the one-click Heroku deployment button.

<Steps>
  <Step title="Click the Deploy button">
    Click the Deploy to Heroku button to start the deployment process:
    
    [![Deploy](https://www.herokucdn.com/deploy/button.svg)](https://heroku.com/deploy?template=https://github.com/coolfool/roxy)
  </Step>

  <Step title="Configure environment variables">
    You'll be prompted to set the following environment variables:

    - **DOMAIN** (required) - The domain to proxy through Heroku. Enter only the hostname (e.g., `example.com`)
    - **KEEP_DYNO_ALIVE** (optional) - Set to `true` to prevent the free dyno from sleeping after 20 minutes. Default is `false`
    - **HEROKU_APP_URL** (optional) - Required if `KEEP_DYNO_ALIVE` is `true`. Enter the full Heroku app URL (e.g., `https://example-app-123.herokuapp.com`)

    <Note>
      If a website redirects to a different domain, use the redirected domain. For example, `google.com` redirects to `www.google.com`, so you should enter `www.google.com` as the `DOMAIN`.
    </Note>
  </Step>

  <Step title="Deploy and access">
    Click the "Deploy app" button and wait for the deployment to complete. Once finished, click "View" to access your Roxy proxy.
  </Step>
</Steps>

## Manual deployment

For more control over the deployment process, you can deploy Roxy manually using the Heroku CLI.

<Steps>
  <Step title="Clone the repository">
    Clone the Roxy repository to your local machine:

    ```bash
    git clone https://github.com/CoolFool/roxy
    cd Roxy
    ```
  </Step>

  <Step title="Login to Heroku">
    Use the Heroku CLI to log in to your account:

    ```bash
    heroku login
    ```
  </Step>

  <Step title="Create a Heroku app">
    Create a new Heroku application and note the generated URL:

    ```bash
    heroku create
    ```

    This will output something like:
    ```
    Creating app... done, ⬢ example-app-123
    https://example-app-123.herokuapp.com/ | https://git.heroku.com/example-app-123.git
    ```
  </Step>

  <Step title="Add buildpacks">
    Roxy requires two buildpacks: Python and a custom Nginx buildpack.

    ```bash
    heroku buildpacks:set heroku/python
    heroku buildpacks:add https://github.com/CoolFool/heroku-buildpack-nginx.git
    ```

    <Info>
      The Python buildpack handles the keep-alive functionality, while the custom Nginx buildpack sets up the reverse proxy with URL rewriting.
    </Info>
  </Step>

  <Step title="Set the DOMAIN variable">
    Configure the domain you want to proxy:

    ```bash
    heroku config:set DOMAIN=example.org
    ```

    Replace `example.org` with your target domain (hostname only, no `http://` or `https://`).
  </Step>

  <Step title="Configure keep-alive (optional)">
    Heroku free dynos sleep after 20 minutes of inactivity. To keep your proxy running continuously:

    ```bash
    heroku config:set KEEP_DYNO_ALIVE=true
    heroku config:set HEROKU_APP_URL=https://example-app-123.herokuapp.com
    ```

    Replace the URL with your actual Heroku app URL from Step 3.

    <Warning>
      Keep-alive functionality sends requests to your app every 5 minutes, which may consume your monthly free dyno hours faster.
    </Warning>
  </Step>

  <Step title="Deploy to Heroku">
    Push the code to Heroku to start the deployment:

    ```bash
    git push heroku main
    ```

    Wait for the build process to complete.
  </Step>

  <Step title="Access your proxy">
    Once deployed, open your Roxy instance:

    ```bash
    heroku open
    ```

    Or visit the URL that was generated when you created the app.
  </Step>
</Steps>

## Testing your proxy

After deployment, test your proxy by visiting your Heroku app URL in a browser. You should see the proxied version of the domain you configured.

For example, if you set `DOMAIN=1337x.to`, visiting your Heroku app URL will display the 1337x.to website through the proxy.

## Viewing logs

To monitor your proxy and troubleshoot any issues, view the application logs:

```bash
heroku logs --tail
```

This will show real-time logs, including:
- Nginx access and error logs
- Keep-alive service status (if enabled)
- Any configuration issues

<Accordion title="Understanding the deployment configuration">
  Roxy uses several key files for deployment:

  **app.json** - Defines the Heroku app configuration:
  ```json
  {
    "name": "Roxy",
    "description": "Deploy Self-Hosted Proxy to unblock websites through Heroku in one click",
    "repository": "https://github.com/CoolFool/Roxy",
    "buildpacks": [
      {
        "url": "heroku/python"
      },
      {
        "url": "https://github.com/CoolFool/heroku-buildpack-nginx.git"
      }
    ]
  }
  ```

  **Procfile** - Defines the process to run:
  ```
  web: bash config.sh
  ```

  **config.sh** - Startup script that initializes the keep-alive service:
  ```bash
  #!/bin/bash
  set -e
  set -o pipefail

  if [[ "${KEEP_DYNO_ALIVE,,}" == "true" ]];then
      nohup python keep_alive.py &
      echo "Heroku dyno will be kept alive"
  else
      echo "Heroku FREE dyno will auto-sleep after 20 minutes"
  fi

  bin/start-nginx-solo
  ```
</Accordion>

## Next steps

Now that you have Roxy running, you can:

- Configure it to proxy different domains by updating the `DOMAIN` environment variable
- Monitor usage and performance through Heroku's dashboard
- Explore the source code to understand how URL rewriting works

<Note>
  Roxy works best with common websites. Complex sites that require multiple domains may not function properly as they need individually written custom rewrite rules.
</Note>
