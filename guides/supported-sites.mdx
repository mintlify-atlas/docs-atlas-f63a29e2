Roxy is designed to proxy simple websites effectively. This guide helps you understand which types of sites work best and how to determine if a site is compatible.

## What works best

Roxy works best with **simple websites** that:

- Operate on a single domain
- Don't require resources from multiple different domains
- Have straightforward HTML/CSS/JavaScript architecture
- Don't use complex client-side routing or dynamic resource loading

<Info>
  Simple content sites, forums, and basic web applications typically work well with Roxy.
</Info>

- `1337x.to`

- `thepiratebay.org`

<Note>
  These sites represent typical use cases for Roxy and demonstrate the types of websites that work well.

## Why complex sites may break

Complex websites often break when proxied through Roxy because they:

### Require multiple domains

Many modern websites load resources from various domains:
- CDN networks for images and assets
- Third-party analytics services
- External API endpoints
- Separate domains for media content

Each of these domains would need **custom rewrite rules** to work correctly through the proxy.

### Use dynamic resource loading

Sites with complex JavaScript that dynamically constructs URLs or loads resources may not work correctly because:
- URLs generated by JavaScript may not be properly rewritten
- WebSocket connections may fail
- Client-side routing may break

<Warning>
  Roxy does not include custom rewrite rules for individual sites. Complex sites requiring such rules will likely not function correctly.
</Warning>

## Testing new sites

When testing if a new site will work with Roxy, follow these steps:

### 1. Configure the correct domain

Ensure you're using the correct domain, accounting for redirects:

```bash
heroku config:set DOMAIN=example.com
```

<Tip>
  If a site redirects (e.g., `example.com` to `www.example.com`), use the final redirected domain.
</Tip>

### 2. Check browser console

Open your browser's developer tools (F12) and check the Console tab for:

- Failed resource requests (404 errors)
- CORS errors
- Mixed content warnings
- JavaScript errors related to URL generation

### 3. Inspect network requests

In the Network tab, look for:

- Requests to multiple different domains
- Failed requests to external resources
- Redirects that aren't being followed correctly

### 4. Test basic functionality

Navigate through the site and test:

- Page navigation and links
- Image and asset loading
- Forms and user interactions
- Any site-specific features

## Identifying sites that need custom rewrite rules

A site likely needs custom rewrite rules if you observe:

<Accordion title="Resources loading from multiple domains">
  If the browser console shows requests to many different domains beyond the main site domain, these requests won't be properly proxied.
  
  **Example:**
  ```
  Failed to load resource: https://cdn.example.com/script.js
  Failed to load resource: https://static.example.net/style.css
  Failed to load resource: https://api.example.io/data.json
  ```
</Accordion>

<Accordion title="Broken images or stylesheets">
  If images, CSS, or other assets fail to load, the URLs may be constructed in a way that Roxy can't automatically rewrite.
</Accordion>

<Accordion title="JavaScript errors about undefined URLs">
  If JavaScript throws errors about undefined or malformed URLs, the site's code may be incompatible with the proxy environment.
</Accordion>

<Accordion title="Features that use WebSockets or real-time connections">
  Sites with real-time features (chat, live updates, etc.) often use WebSocket connections that may not work through the proxy.
</Accordion>

## Importance of correct DOMAIN configuration

The `DOMAIN` environment variable is critical for proper redirect handling. Here's why:

### Redirects must match

If your target site redirects to a different domain and your `DOMAIN` setting doesn't match, the proxy will break.

**Example scenario:**

1. You set `DOMAIN=google.com`
2. Google redirects all requests to `www.google.com`
3. Roxy is configured for `google.com` but receives content from `www.google.com`
4. Links and resources break because of the domain mismatch

**Solution:**

```bash
# Use the final redirected domain
heroku config:set DOMAIN=www.google.com
```

<Warning>
  Always test the domain in your browser first and note where it redirects before configuring Roxy.
</Warning>

## When Roxy isn't the right choice

Consider alternative solutions if:

- The site requires multiple domains to function
- You need WebSocket support
- The site uses complex client-side routing (SPA frameworks)
- You need to proxy authenticated sessions with complex cookie handling
- The site has anti-proxy measures in place

For such cases, you may need a more sophisticated proxy solution with custom rewrite rules for the specific site.
